## Context

当前游戏基于 Vue 3 + Vite + Phaser.js + Pinia 技术栈，采用组件视图切换模式（GameApp.vue 通过 `currentView` 控制显示哪个子组件）。已有冒险模式（adventure/battle）和商店模式（shop/cashier）两种玩法，共享题库系统（questionStore）、经济系统（gameStore 的 coins/exp）、成就系统和存档系统（LocalStorage）。

项目处于早期阶段，大部分任务尚未完成。新增玩法需要在现有架构上扩展，保持松耦合。

## Goals / Non-Goals

**Goals:**
- 新增三种独立子玩法（速算、工坊、卡牌），统一通过挑战中心入口访问
- 所有玩法产出金币，可在商店消费，形成经济闭环
- 支持 1~6 年级差异化内容和难度
- 新增钻石作为稀有货币，控制获取节奏（周产出 2-5 个）
- 工坊制品可上架商店，后台异步销售，玩家可切出做其他事
- 卡牌对战支持三档 AI 难度（简单/中等/复杂），默认按年级推荐
- 虚拟排行榜提供动态难度挑战感

**Non-Goals:**
- 不做后端/在线多人功能（纯 LocalStorage 离线方案）
- 不做实时推送通知（使用应用内红点/弹窗）
- 不做复杂的卡牌平衡性系统（面向小学生，策略深度有限）
- 不修改现有冒险模式和商店模式的核心逻辑（向后兼容）

## Decisions

### 1. 视图路由：扩展 GameApp 的 currentView 模式

**决策**: 沿用现有的 `currentView` ref 控制视图切换，新增 `challenge`、`speedChallenge`、`workshop`、`cardBattle`、`cardCollection`、`leaderboard` 视图状态。

**理由**: 现有架构已经成熟，不需要引入 Vue Router 增加复杂度。

**替代方案**: 引入 Vue Router — 增加复杂度，对单页游戏应用收益有限。

### 2. 挑战中心作为中间层视图

**决策**: GameHall.vue 作为挑战中心页面，不直接参与答题逻辑，仅作为导航入口展示三个玩法卡片和快捷状态（如速算最佳成绩、工坊待售数量）。

**理由**: 解耦入口和玩法逻辑，便于后续扩展更多子玩法。

### 3. 经济系统扩展：gameStore 新增 gems 和 stars 字段

**决策**: 在现有 gameStore 的 state 中新增 `playerGems`（钻石）和 `playerStars`（星星）字段，新增 `addGems()`、`spendGems()` 方法。

**理由**: 金币已经通过 gameStore 管理，钻石作为第二货币放在同一 store 便于统一管理和存档。

### 4. 卡牌数据模型：只存 ID 和数量

**决策**: 卡牌数据库定义在 `config/cards.js` 中，玩家收藏只存储 `{ cardId, quantity }` 而非完整卡牌数据。

**理由**: LocalStorage 容量有限（5MB），60 张卡牌 × 收藏数量 × 卡组数据如果存完整对象会很大。只存引用可将存档控制在 50KB 以内。

### 5. AI 策略实现：启发式规则而非博弈树

**决策**:
- **简单**: 随机出牌
- **中等**: 基于 HP 阈值的优先级规则（HP<30% 防御，HP>70% 攻击，有回复就用）
- **复杂**: 简单评估函数（对每张候选牌计算期望收益 = 伤害/回复/防御值 × 稀有度系数），选最高分

**理由**: 完整博弈树对浏览器环境过重，且小学生不需要那么深的 AI。启发式规则足够提供差异化体验。

### 6. 工坊后台销售：时间戳差值计算

**决策**: 上架时记录 `listedAt` 时间戳，玩家返回时根据 `Date.now() - listedAt` 计算经过的"游戏时间"，按每 30-60 秒模拟一位顾客的频率结算销售结果。

**理由**: 支持玩家切出后回来看到销售结果，不需要后台进程或定时器。

**风险**: 如果玩家长时间不打开（如几天），可能积累大量销售。设置上限（每次最多结算 10 笔交易）。

### 7. 虚拟排行榜：动态分数生成

**决策**: 虚拟玩家数据存储在 `config/leaderboard.js` 中，包含名字池和分数范围配置。玩家首次进入时生成一组虚拟玩家，分数基于当前年级水平在合理范围内波动。玩家打破记录后，部分虚拟玩家分数"增长"（×1.05~1.15），保持"差一点就能超越"的紧张感。

### 8. 通知系统：轻量级 Store + Navbar Badge

**决策**: 新增 `notificationStore` 管理通知队列，在 `GameNavbar.vue` 上显示红点 badge。通知类型包括：工坊售出、卡牌获得、成就解锁。

**理由**: 不需要独立的通知中心页面，红点 + 弹窗足够。保持轻量。

### 9. Store 间通信：通过 gameStore 中转

**决策**: 各玩法 store（cardStore、workshopStore、speedChallengeStore）保持独立，但涉及金币/钻石变更时调用 gameStore 的方法，而非直接修改 state。

**理由**: 保持松耦合，避免 store 间循环依赖。gameStore 作为经济系统的单一入口。

### 10. 年级适配：配置驱动而非硬编码

**决策**: 每个玩法的配置文件中定义年级对应的内容范围（如卡牌按年级分组、速算题目类型按年级过滤），通过 settingsStore.grade 动态加载。

**理由**: 便于后续调整年级内容，不需要修改业务逻辑代码。

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LocalStorage 存档膨胀 | 存档过大导致读写慢或超限 | 只存 ID 引用，定期清理过期数据，预估 < 50KB |
| 工坊离线销售计算偏差 | 玩家长时间未上线导致销售结果不合理 | 设置单次结算上限（10 笔），超过部分顺延 |
| AI 复杂策略性能 | Hard AI 评估函数在低端设备上卡顿 | 限制手牌数量（最多 5 张），评估函数保持 O(n) |
| 视图状态管理复杂度 | currentView 状态过多导致 bug | 使用状态机模式管理视图转换，禁止非法跳转 |
| 卡牌平衡性 | 某些卡牌过强/过弱影响体验 | 初始版本保持卡牌效果简单，后续根据反馈调整 |
| 年级跨度过大 | 1 年级和 6 年级内容差异导致维护困难 | 严格按年级分组配置，共享基础逻辑 |
| 多玩法并行开发冲突 | 多人开发时文件冲突 | 每个玩法独立目录，共享代码通过 utils 层 |
